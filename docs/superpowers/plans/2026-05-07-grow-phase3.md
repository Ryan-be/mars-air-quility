### Going-offline event writer

Modify `mlss_monitor/grow/ws_registry.py` to take a callback or DB path so `register` can INSERT a `grow_errors` row with `kind='online'` (resolves any open `kind='offline'` row) and `unregister` can INSERT with `kind='offline'`.

The cleanest approach: don't put DB access into `WSRegistry` itself (it's a pure connection map). Instead, add the DB write at the call site in `mlss_monitor/routes/api_grow_ws.py::_connection_handler`:

```python
async def _connection_handler(ws, path: str, registry):
    unit_id = _parse_path(path)
    # ... validation ...
    
    registry.register(unit_id, ws)
    _record_connection_event(unit_id, "online")  # NEW
    log.info("grow unit %s connected", unit_id)
    
    try:
        # ... existing receive loop ...
    finally:
        registry.unregister(unit_id)
        _record_connection_event(unit_id, "offline")  # NEW
        log.info("grow unit %s disconnected", unit_id)


def _record_connection_event(unit_id: int, kind: str) -> None:
    """Write a grow_errors row capturing the connection state change.
    
    For kind='online': also resolves the most recent open kind='offline'
    row for this unit. For kind='offline': just inserts. Best-effort —
    a DB error here must not tear down the WS handler.
    """
    try:
        conn = sqlite3.connect(DB_FILE, timeout=5)
        try:
            now = datetime.utcnow()
            if kind == "online":
                conn.execute(
                    "UPDATE grow_errors SET resolved_at=? "
                    "WHERE unit_id=? AND kind='offline' AND resolved_at IS NULL",
                    (now, unit_id),
                )
            conn.execute(
                "INSERT INTO grow_errors "
                "(unit_id, timestamp_utc, severity, kind, message) "
                "VALUES (?, ?, ?, ?, ?)",
                (unit_id, now, "warning" if kind == "offline" else "info",
                 kind, f"unit {kind}"),
            )
            conn.commit()
        finally:
            conn.close()
    except Exception as exc:
        log.warning("connection_event write failed for unit %s: %s", unit_id, exc)
```

## Task 2: Firmware reports firmware_version + uptime + buffer_size

### Firmware changes

`grow_unit/src/mlss_grow/service.py` (or wherever capabilities is emitted at boot):

Read firmware version from package metadata:
```python
from importlib.metadata import version, PackageNotFoundError
def _get_firmware_version() -> str:
    try:
        return version("mlss_grow")
    except PackageNotFoundError:
        return "dev"
```

Include in the `capabilities` message envelope (NOT inside any individual capability — it's per-unit metadata):
```python
caps_payload = {
    "capabilities": [...],
    "firmware_version": _get_firmware_version(),
    "uptime_s": time.monotonic() - service_start_time,
}
```

For ongoing uptime + buffer_size: add fields to the `telemetry` message payload that the safety_loop emits:
```python
telemetry_payload = {
    # existing fields ...
    "uptime_s": time.monotonic() - service_start_time,
    "buffer_size": buffer.size(),
}
```

### Server-side changes

Update `handle_capabilities` in `mlss_monitor/grow/handlers.py` to UPDATE `grow_units.firmware_version` from the message envelope.

Update `handle_telemetry` to UPDATE `grow_units.last_uptime_s` and `last_buffer_size` from the telemetry payload.

### Contracts changes

Update `mlss_contracts/ws_messages.py`:
- `CapabilitiesMessage`: add optional `firmware_version: str | None`, `uptime_s: float | None`
- `TelemetryMessage`: add optional `uptime_s: float | None`, `buffer_size: int | None`

Backward compat: optional with None default, so existing tests that don't supply them still pass.

## Task 3: GET `/api/grow/units/<id>/diagnostics` consolidated endpoint

The Diagnostics tab fetches one consolidated payload rather than 4 separate calls. Returns:

```json
{
  "firmware_version": "1.2.3",
  "uptime_s": 12345.6,
  "buffer_size": 0,
  "connection_log": [
    {"id": 12, "timestamp_utc": "...", "kind": "online", "resolved_at": null},
    {"id": 11, "timestamp_utc": "...", "kind": "offline", "resolved_at": "..."}
  ],
  "sensor_sanity": [
    {"channel": "soil_moisture", "last_seen_at": "...", "minutes_ago": 0.5,
     "is_stale": false, "stale_threshold_min": 5},
    {"channel": "ambient_lux", "last_seen_at": null,
     "minutes_ago": null, "is_stale": true, "stale_threshold_min": 5}
  ],
  "open_errors": [
    {"id": 9, "timestamp_utc": "...", "severity": "warning",
     "kind": "sensor_degraded", "message": "...", "subject_sensor": "ambient_lux"}
  ]
}
```

`connection_log`: last 20 entries from `grow_errors` where `kind IN ('online', 'offline')`, ORDER BY id DESC.

`sensor_sanity`: per-capability, derive `minutes_ago` from `last_seen_at`, `is_stale = minutes_ago > 5` (configurable via app_setting `grow_sensor_stale_threshold_min`).

`open_errors`: all `grow_errors` rows for this unit where `resolved_at IS NULL` (excluding online/offline meta-events).

## Self-review notes

- Tasks are mostly independent — could parallelise Tasks 4-6 if needed, but sequential is safer
- Schema changes are additive (3 new nullable columns); no migration risk
- `watering_state.json` is a small, isolated firmware add — could be Task 1 or Task 7 (does't depend on any other task)
- Storage warning depends on `grow_disk_warn_pct` (already seeded as Phase 1 — verify it's there)
- The Diagnostics tab's "Move token rotation here" doesn't break Configure tab — keep both for now (the rotator is identical), deprecate the Configure-tab placement later
- Decommission is a soft-delete (`is_active=0`); doesn't drop telemetry history, doesn't cascade-delete anything
- `clear_buffer` is destructive — the operator could lose un-replayed data if they click during an outage. Confirm modal MUST warn explicitly
